## 一、产品概述

**产品名称**：计数助手

**目标用户**：市集摆摊摊主，需要在售卖过程中快速记录不同产品的销售数量

**核心场景**：
- 摊主在忙碌的售卖中，单手快速点击产品按钮 +1 计数
- 收摊后查看当天收入、导出 CSV 记账
- 查看历史销售记录

**分发策略**：
- 免费版通过微信群/朋友圈传播，功能完整但产品数量受限
- 专业版通过激活码解锁，微信社群直销
- 后续上架 App Store 扩展全球市场

| 定位 | 让摊主免费体验全部核心功能，觉得好用自然会分享和升级 |

### 专业版

| 项目 | 说明 |
|------|------|
| 产品数量上限 | 20 个 |
| 网格模板 | 5/10/15/20 宫格全部可用 |
| 其他功能 | 与免费版完全一致 |
| 获取方式 | 输入激活码解锁 |
| 定价 | 待定（通过微信社群销售） |

### 设计原则

- **免费版必须好用**：限制仅在产品数量，其余功能完整。用户用着舒服才会传播。
- **升级路径自然**：生意越好、产品越多，越需要专业版。限制点在用户成长路径上。
- **不拆分文件**：同一个 `index.html`，通过激活状态切换行为。方便维护和传播。

## 三、激活码系统

### 格式

```
CP-XXXX-XXXX-XXXX
```

示例：`CP-A3K9-M7W2-Q5R1`

字符集：大写字母 + 数字（去掉易混淆的 0/O/I/L/1），共 28 个字符可用。

### 验证方式

1. 预设 20 个激活码，将其 SHA-256 哈希值内置于 JS 代码中
2. 用户输入激活码 → Web Crypto API 计算 SHA-256 → 与内置哈希列表比对
3. 匹配成功 → localStorage 写入激活状态，解锁专业版
4. 每个激活码可重复使用（初期不做设备绑定，信任传播）

### 激活码管理

- 开发者预先用脚本生成 20 个激活码 + 对应哈希
- 激活码存放在本地管理文件（不提交到公开仓库）
- 用户购买后，通过微信手动发送激活码

### 安全性说明

- 前端验证无法做到绝对安全，但哈希存储可防止普通用户直接查看源码获取激活码
- 预设固定码方式最简单，适合初期微信社群销售
- 后续 App Store 版本替换为 IAP 验证，无需激活码

## 四、功能需求详述

### 4.1 激活入口

**位置**：设置页 → 新增"版本信息"区块（外观设置下方）

**界面元素**：
- 当前版本标识（免费版灰标 / 专业版金标）
- 免费版显示：激活码输入框 + "激活专业版"按钮
- 专业版显示："已激活专业版 ✓" + 使用的激活码（脱敏显示，如 `CP-A3***-***5R1`）

**交互**：
- 输入激活码，自动转为大写，自动按 4 位分组
- 点击激活 → 验证 → 成功弹 toast "已解锁专业版！" / 失败弹 toast "激活码无效"
- 激活状态持久化到 localStorage，刷新不丢失

### 4.2 产品数量限制

**限制规则**：免费版最多添加 5 个产品

**触发场景**：

| 场景 | 行为 |
|------|------|
| 添加第 6 个产品 | 弹出升级提示弹窗，阻止添加 |
| 切换到 10/15/20 宫格模板 | 弹出升级提示弹窗，阻止切换 |

**升级提示弹窗**：
- 标题："专业版功能"
- 正文：说明免费版限制（最多 5 个产品 / 仅 5 宫格），引导购买激活码
- 按钮：「输入激活码」（跳转到设置页版本区块）、「暂不升级」（关闭弹窗）
- 联系方式：预留微信联系方式文案

### 4.3 顶部栏状态

- 日期左侧显示版本标记
- 免费版：灰色小圆点 + "免费版" 文字
- 专业版：金色小圆点 + "PRO" 文字

### 4.4 模板选择器

- 免费版：5 宫格正常可选，10/15/20 宫格显示锁定图标
- 专业版：全部可选
- 点击锁定项 → 弹出升级提示

### 4.5 激活状态管理

- `localStorage` 中存储：
  - `counter-pro-activated`: `true/false`
  - `counter-pro-code`: 使用的激活码（脱敏前完整存储，用于显示）
- 清除浏览器数据会导致激活状态丢失，需重新输入
- 初始化时从 localStorage 读取，无需网络请求

## 五、技术方案

### 技术栈

- 纯 HTML/CSS/JavaScript，单文件，无框架依赖
- SHA-256 哈希：Web Crypto API（`crypto.subtle.digest`）
- 存储：localStorage
- 部署：GitHub Pages（免费静态托管）

### 代码结构

激活相关代码集中在以下区域：

```
index.html
├── CSS: 版本徽标、升级弹窗、锁定模板样式
├── HTML: 版本信息区块、升级弹窗、顶部栏徽标
└── JS:
    ├── 激活码哈希列表（常量）
    ├── isPro() — 判断是否专业版
    ├── checkActivation() — 初始化时读取激活状态
    ├── activatePro(code) — 验证并激活
    ├── showUpgradeModal() — 显示升级弹窗
    └── 修改 btnAdd / templateGrid 逻辑加入限制检查
```

### 激活码生成脚本

在项目内创建 `.claude/tools/generate-codes.html`（本地工具，不提交到 git）：
- 输入：生成数量
- 输出：激活码明文列表 + SHA-256 哈希列表
- 用途：开发者本地生成激活码，哈希值复制到 index.html，明文保存备用

### 兼容性

- Web Crypto API 需要 HTTPS 或 localhost 环境
- GitHub Pages 默认 HTTPS，满足要求
- 本地开发用 `file://` 协议时 crypto.subtle 不可用，需降级处理（见风险点）

---

## 六、风险点与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| `file://` 下 crypto.subtle 不可用 | 本地打开无法激活 | 检测环境，file 协议下降级为明文比对 + 控制台警告 |
| 激活码被分享 | 一个码多人使用 | v2 阶段接受此风险，v3 可加设备绑定 |
| 清除浏览器数据丢失激活 | 用户需重新输入 | 激活时提示用户保存激活码，专业版显示激活码供回填 |
| localStorage 被篡改 | 可手动设 proActivated=true | 前端无法防御，但目标用户（摊主）无此动机 |
